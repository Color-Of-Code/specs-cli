# Roles

The specs platform is used by a small set of recurring roles. These roles are
operational, not organisational: one person may hold several of them in the
same repository.

## Role map

| Role             | Primary responsibility                                                     | Typical workflows                                                             |
| ---------------- | -------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Any user         | Use the installed engine successfully in their local environment           | Run `specs doctor`; initialise VS Code tasks                                  |
| Project owner    | Stand up or onboard a host repository so the team can work in it           | Bootstrap a new host; initialise an existing repo                             |
| Spec author      | Create and maintain requirements, features, components, APIs, and services | Open change requests; scaffold artifacts; lint and format specs; verify links |
| Reviewer         | Check that proposed model changes are structurally sound and traceable     | Verify links; inspect traceability graphs; review baselines                   |
| Stakeholder      | Consume traceability output to understand coverage and gaps                | View traceability graphs                                                      |
| Component owner  | Keep recorded component baselines aligned with tracked upstream repos      | Verify and refresh baselines                                                  |
| Platform admin   | Manage user-level framework registry entries for teams or machines         | Add, remove, and list framework registry entries                              |
| Framework author | Create, evolve, and distribute framework content                          | Seed a new framework; update consumers; author templates and process docs     |

## Role descriptions

### Any user

An any user is any contributor who needs the engine and editor integration to
work on their machine. This role is not tied to authoring or reviewing. Its job
is to confirm the local install, framework resolution, and editor wiring are in
good shape before deeper work starts.

Typical actions:

- Run `specs doctor` when a command behaves unexpectedly.
- Run `specs vscode init` when local VS Code tasks are needed.

### Project owner

A project owner prepares the host repository so other contributors can start
working without manual setup. This role chooses the framework source and
consumption mode, writes the initial `.specs.yaml`, and optionally seeds local
editor tasks.

Typical actions:

- Run `specs bootstrap` in a new repository or subtree.
- Run `specs init` in an existing repository.
- Decide whether the host uses a managed framework, local checkout, vendor
  snapshot, or submodule layout.

### Spec author

A spec author produces and edits the actual model content. This is the main
day-to-day user of the platform. The role works inside change requests, creates
new artifacts from templates, keeps markdown clean, and ensures traceability is
correct before review.

Typical actions:

- Run `specs cr new` and `specs cr drain`.
- Run `specs scaffold` for requirements, features, components, APIs, and
  services.
- Run `specs format`, `specs lint`, and `specs link check` during iteration.

### Reviewer

A reviewer evaluates whether a proposed change request is coherent, complete,
and traceable. This role does not need to author content directly, but it does
need tooling that exposes structural gaps, missing links, and baseline drift.

Typical actions:

- Run `specs link check` to confirm bidirectional traceability.
- Generate or inspect traceability visualisations during review.
- Read baseline updates to confirm the spec still points at the intended
  upstream component revision.

### Stakeholder

A stakeholder is a consumer of the model rather than an editor of it. This role
uses the platform to understand scope, coverage, and dependency structure across
requirements, features, and components.

Typical actions:

- View traceability diagrams generated by `specs visualize traceability`.
- Use the graph to spot uncovered requirements, orphaned implementers, or broad
  areas of impact.

### Component owner

A component owner is responsible for the relationship between a component spec
and the upstream repository revision it describes. This role keeps the recorded
baseline SHAs current so reviewers can see exactly what implementation snapshot
the spec matches.

Typical actions:

- Run `specs lint --baselines` to detect drift.
- Run `specs baseline update` to refresh the baselines table.
- Review the resulting diff before it is committed.

### Platform admin

A platform admin manages the per-user framework registry that maps friendly
names to framework sources. This role reduces setup friction for teams and
standardises which framework entries contributors select during `init` and
`bootstrap`.

Typical actions:

- Run `specs framework add` to register shared framework entries.
- Run `specs framework list` to inspect configured entries.
- Run `specs framework remove` to retire obsolete entries.

### Framework author

A framework author creates, evolves, and distributes framework content:
templates, process docs, skills, agents, and lint configuration. This role
covers both authoring the framework repository and pushing updates to host
projects that consume it.

Typical actions:

- Run `specs framework seed --out <dir>` to create a new framework skeleton.
- Author and version framework contents in a dedicated repository.
- Publish the framework and register it so host projects can consume it.
- Run `specs framework update` to roll a newer ref into a host project.
- Run `specs doctor` and `specs lint` after the update to surface follow-on
  fixes.

## How the roles relate

Most repositories mainly involve three roles: project owner, spec author, and
reviewer. The remaining roles appear when the platform is being operated or
extended:

- Any user covers local setup and diagnosis.
- Component owner focuses on baseline integrity for tracked repos.
- Platform admin manages registry configuration across machines or teams.
- Framework author produces and distributes framework content that project
  owners consume.
